# 265 Frontend Build Contract

Purpose: define what frontend is responsible for in the public-site clean-build run.

## Frontend mission
Build the public site as one coherent operating surface.

That means:
- one company
- one calm intelligence
- one unmistakable surface
- one route spine
- many page families with distinct burdens

## Frontend must understand
### Mechanics
- what moves
- what breathes
- what reacts
- what stays still
- what transitions are allowed

### Flow
- how a person arrives
- how the page reveals itself
- how burden passes from route to route
- how CTA continuity works

### Hierarchy
- what the hero must do
- what bridges must do
- what sections must do
- what lower-spine proof must do
- what pages should not try to do

### Philosophy
- state before story
- proof over performance
- architecture of grace
- one system, many rooms, one conductor
- old homepage is reference, not doctrine

## Frontend outputs
For each route, frontend should be able to specify:
- module stack
- component stack
- motion stack
- asset role needs
- copy envelope needs
- evidence capture points
- unresolved doctrine dependencies

## Frontend should ask
- what is the page burden
- what is the dominant module
- what is the dominant mechanic
- what can be removed because another route already carries it
- what must remain conditional because doctrine is not closed

## Frontend should not ask too late
- what the page is for
- whether this route is consumer-facing or expert-facing
- whether hero has to prove the whole company
- whether an asset is decorative or structural

Those should already be answered by the marketing packet.
