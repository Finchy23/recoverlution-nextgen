# 263 Route Composition Implementation Notes

Purpose: give design/frontend a practical route-by-route build note for the first clean-build targets.

## Homepage
- preserve one dominant hero field
- do not start with multiple parallel information blocks
- if a section cannot be tied to a Family Window or Trust Proof Band burden, it probably should not be on the homepage
- the first three sections should not each try to explain the whole product

## `/product`
- structure around organism/system coherence, not modality inventory
- avoid four-equal-column feature logic
- supporting evidence should appear after the organism frame, not before it

## `/method`
- structure around the loop of change, not a generic “how it works” page
- avoid mixing scientific explanation and product walkthrough in the same block
- let one Architecture Reveal carry the hidden-complexity burden

## `/proof`
- lead with proof logic, not general category story
- avoid metric theater
- if the page starts to look like analytics, it is wrong

## `/authority`
- lead with why the authority model is different
- avoid visual language that feels institutional or dashboard-like
- use supporting evidence to strengthen the claim, not to simulate scientific certainty

## Shared implementation guardrails
- route structure remains fixed
- module burden comes before layout novelty
- assets follow route burden
- copy follows route burden
- unresolved naming/media/auth doctrine stays conditional and must not be hardened in implementation
