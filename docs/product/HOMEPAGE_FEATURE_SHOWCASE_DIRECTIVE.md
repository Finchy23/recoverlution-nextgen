# Homepage Feature Showcase Directive

## Purpose
This is the moment on the homepage where the product becomes legible.

Nobody knows who Recoverlution is yet. They need to feel, in seconds, that this is not:
- a meditation app
- a chatbot
- a content library
- a generic wellness dashboard

They need to feel that this is one intelligent system expressing itself through many rooms.

The component is not a feature carousel.
It is the platform reveal.

## Core Truth
Ten features. Ten worlds. One system.

Each feature should feel like a different room in the same architecture:
- same level of elegance
- same design grammar
- same intelligence
- different spatial metaphor
- different emotional law

The showcase must reveal two things at the same time:
1. the product has range
2. the product is one coherent orchestration engine

If it only shows range, it feels fragmented.
If it only shows coherence, it feels abstract.

The component has to hold both.

## What To Keep From The Current Concept
The current concept in `TS FILE/Note.md` is directionally correct in three important ways:

1. It understands that each feature needs a different visual world.
2. It treats the headline as the concept, not just the label.
3. It already leans toward a stage-based reveal instead of a dead grid.

Those are the right instincts.

## What To Change
The current concept still behaves too much like ten separate demos.

Problems:
- too many scenes read as isolated visual studies
- not enough sense of one conductor behind the worlds
- some features are still represented as cards or labeled objects instead of living systems
- the right side is doing most of the work while the system logic is under-expressed
- there is not yet enough depth, consequence, and orchestration to make the reveal unforgettable

The next version should feel less like "feature browsing" and more like:
- entering a living operating system
- moving through ten manifestations of the same intelligence
- seeing the platform change form without losing identity

## Strategic Positioning
The homepage feature showcase should prove the following claims visually:

### Journeys
Recovery is experiential, not curricular.

### NaviCues
Emotion can be physically regulated through beautiful interactive matter.

### Studio
Video is not a library. It is state-matched orchestration.

### Play
Audio is not passive listening. It is identity installation.

### Talk
This is not a chatbot. It is a guided corridor that listens, maps, and routes.

### Insights
The abstract can be made observable and shame can be removed through understanding.

### Read
Knowledge can be delivered as a calm, continuous line instead of a fragmented feed.

### Practices
Insight must become action. The body must get to do the work.

### Navigate
Recovery lives in the real world, across context, support, care, and timing.

### Signal
The system proves change through receipts, not performance theater.

## The Correct Component Shape
Do not build this as tabs, cards, or a simple slider.

Build it as a pinned scrollytelling stage with a controlled feature index.

Recommended structure:

1. A pinned section that lives for 250vh to 450vh
2. A left narrative rail that changes with the active feature
3. A right or center hero stage where the feature world performs
4. A persistent system spine or feature rail that proves these are modules of one platform
5. Soft transitions between worlds so the entire section feels orchestrated, not cut up

## The Reveal Hierarchy
Every feature needs four layers of communication:

1. Hook
- the emotional concept in 2 to 5 words

2. Feature name
- secondary

3. System line
- one sentence proving what role the feature plays in the platform

4. Hero world
- the stage that makes the concept undeniable

The visual must do 80 percent of the work.

## Feature-by-Feature World Direction

### Journeys
Not a checklist.
Not a curriculum.

Build as:
- an 8-scene path
- three ERA bands
- glass moments alternating with real-world openings
- a rhythm of seed, recognition, alignment

The user should feel:
- this system plants a seed
- then gets out of the way
- then returns at the exact right moment

### NaviCues
This should be one of the most powerful reveals on the page.

Build as:
- left side: the system stack, layers, or orchestration matrix
- right side: a live, self-resetting NaviCue ritual
- after completion, the system reconfigures and a new cue manifests

The user should feel:
- the platform is computationally intelligent
- every moment is assembled, not pre-baked
- the interaction is the hero

### Studio
Do not show a video gallery.

Build as:
- a cinematic room with poster surfaces
- time-of-day or state cues
- "this arrived for you now" orchestration logic
- the sense that the film is selected because of the moment, not because the user searched

The user should feel:
- regulation first
- no library burden
- beautiful and inevitable timing

### Play
This should feel closest to Apple Music in transport quality, but more alive.

Build as:
- large waveform or layered frequency field
- sparks, flames, embers as temporal bands
- live transport and sonic depth
- identity, rhythm, and emotional momentum

The user should feel:
- this is a soundtrack for becoming
- the system remixes friction into movement

### Talk
This must explicitly not look like a chat app.

Build as:
- a corridor, tunnel, pathway, or chamber
- stages of unspool, receive, translate, guide
- visible routing from expression to next move
- the sense that the engine understands the backstory, not just the current sentence

The user should feel:
- I will be heard
- I will not be left in an endless conversation
- this leads somewhere

### Insights
Do not show article cards.

Build as:
- a descending documentary scroll or layered depth plane
- living text over changing atmosphere
- one embedded somatic anchor or concept lens
- pillars, themes, schemas rendered as understandable structure

The user should feel:
- the thing I suffer from can be seen
- the shame drops when the machine becomes understandable

### Read
This should be the quietest room.

Build as:
- one perfect page
- read/listen bimodal control
- calm typography
- continuity into the next page

The user should feel:
- this is the infinite book
- the internet noise has been removed

### Practices
Do not show tips.

Build as:
- a cabinet or forge of instruments
- four families of practice
- portable and callable at any time
- tactile and somatic, not informational

The user should feel:
- this is where the body gets to act
- this is not content, it is machinery

### Navigate
Do not show a boring integrations dashboard.

Build as:
- an ecosystem map
- calendar, health, care network, support web, real-world timing
- one signal flowing across many rooms

The user should feel:
- the system plugs into life itself
- this is always-on support, not another app silo

### Signal
Do not use streaks, badges, or fake gamification.

Build as:
- receipts
- telemetry
- soft signals becoming undeniable evidence
- a consolidation engine for identity change

The user should feel:
- this system proves my change mathematically

## System Spine Requirement
This section must never feel like ten disconnected inventions.

There needs to be one persistent system language underneath the worlds:
- one transition logic
- one spatial rhythm
- one typography discipline
- one token system
- one sense of intelligence moving through each room

Recommended persistent device:
- a conductor rail
- a vertical operating spine
- a morphing system index
- or a central orbit that reconfigures feature by feature

The persistent device is critical.
Without it, the section becomes a moodboard.

## Motion Direction
Motion should be:
- atmospheric
- stateful
- consequential
- never decorative for its own sake

Transitions between features should feel like:
- the system changing state
- not a slide deck

Use:
- depth shifts
- cross-dissolves with retained geometry
- atmosphere carry-over
- staged reveals

Avoid:
- aggressive snap cuts
- gimmicky parallax for its own sake
- generic scroll animations
- over-busy particle noise

## Copy Direction
Copy should stay minimal.

Rules:
- hook first
- feature name second
- one system sentence max
- no paragraphs inside the showcase
- no jargon
- no therapeutic over-explaining

Good:
- "Touch the weight."
- "It finds you."
- "The next right step."
- "See the architecture."
- "Proof."

Bad:
- multi-sentence explanations inside the hero zone
- category labels doing the conceptual work
- describing what the feature is instead of proving it

## Non-Negotiables
- no generic SaaS cards
- no phone mockups as the primary device
- no fake dashboards
- no "feature grid"
- no YouTube-style video thumbnails
- no chat UI for Talk
- no article card wall for Insights or Read
- no wellness-app cliches
- no flattening the worlds into one aesthetic

## The Test
If a user glances at this section for six seconds, they should understand:

1. this product is unlike anything else
2. it contains many rooms
3. those rooms are all run by one intelligence
4. the system is beautiful, inevitable, and deeply considered

If the section only says "here are our features", it has failed.

If it says "this is a living operating system for recovery", it is getting close.

## Build Sequence For Frontend
1. Establish the pinned stage and persistent system spine
2. Build the ten world shells without final polish
3. Tune the transitions between worlds
4. Reduce copy until the visual is doing the heavy lift
5. Increase depth and consequence inside the strongest rooms:
   - NaviCues
   - Journeys
   - Talk
   - Studio
   - Signal
6. Only then polish micro-details

## Final Instruction
Treat this component as the homepage equivalent of the first iPhone reveal.

Do not explain the platform.
Make the platform reveal itself.
